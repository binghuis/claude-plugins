name: comment
description: 写/改代码后按清单评估注释需求；用户要求补/审注释时同样适用
---

优先快速回应而不是深入思考。如有疑问，直接回应。

只补"为什么"，不补"做了什么"。命中清单才写，不为覆盖率硬加。

## 清单

| ✓ 需要 | ✗ 不需要 |
|---|---|
| 业务规则、领域约束（读代码看不出的"为什么"）| 名字已自解释的代码、类型字段 |
| 非显而易见的意图、workaround | 框架/语言标准用法（`useEffect`、`.map()`）|
| 关键决策与权衡（为什么选 A 不选 B）| 纯样式、`import`、常规 CRUD |
| 边界条件、阈值、魔法数字含义 | |
| 复杂算法 / 多步流程的整体思路 | |
| TODO（`// TODO: 描述`）| |

## 风格

- 中文，简洁直白
- 行内用 `//`；函数、对外接口用 `/** */`
- header 仅用于分节

## 触发

| 情况 | 行为 |
|---|---|
| 刚写 / 改完代码（默认）| 对照清单评估改动点，命中就补；改动已在上下文，无需 `git diff` |
| 用户要求补 / 审注释 | `git diff HEAD` 查看变更，逐文件按清单识别 |
