# 迭代需求的处理

适用场景：基于现有项目的改动需求（占成熟项目需求的大多数）。

## 为什么迭代需求比全新需求难

全新需求（1.0）的参照系是空白，只需让需求自身完整自洽。

迭代需求的"对错"是**相对于现有系统定义的**。一个迭代需求是否完整、合理、有没有坑，取决于现有项目：现在怎么实现的？有没有可复用模块？会不会和现有逻辑冲突？改了影响哪里？——这些光看需求文档永远答不出来，必须结合现有代码库。

## 这意味着的能力要求

迭代需求的结构化，比全新需求多一个核心能力：**理解并参照现有项目的实现**，据此判断需求落在现有系统里哪里通、哪里冲突、哪里缺。

这正是当前 AI 编程工具（Cursor、Claude Code 等）在攻坚的"代码库上下文理解"。

## 务实策略（重要）

**不要自己从零造一个"能理解任意代码库"的系统**——那是业界顶尖工具都还在攻坚的重命题，超出单个 skill 该承担的范围。

正确做法：
- **借助现有 AI 编程工具已有的代码库理解能力**，把本 skill 构建在其之上。
- 专注于"在你这个特定项目里，把需求与代码库的对照做到极致"——这恰恰依赖你对自己项目的深度理解，是别人难复制的护城河。

## 迭代需求额外要产出的内容

在全新需求的产出基础上，迭代需求还要补：

1. **与现有实现的关系**：涉及现有哪些模块？有无可复用实现？会不会和现有逻辑冲突？
2. **影响范围/改动面**：波及哪些现有功能、模块、文件？有无回归风险？
3. **基于现有实现的待澄清项**：很多模糊点不是需求本身模糊，而是"需求和现有实现对不上"。例：需求说"加个登录方式"，但现有登录架构不支持 → 这是必须澄清的冲突点。这类问题只有结合代码库才能发现，价值极高。

## 落地顺序建议

先把全新需求的闭环（需求判定 → 三分 → 产出）在真实项目跑通、验证有效，再向迭代需求延伸。迭代这块是最难、也最有护城河的部分，不要一上来就啃。
