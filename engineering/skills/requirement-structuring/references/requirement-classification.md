# 需求判定

## 判据

属于业务需求：定义"做什么能力 / 业务规则 / 逻辑分支"。

不属于业务需求：描述"长什么样 / 怎么交互 / 怎么实现"。

混合时拆开：业务实质归需求，呈现归设计。

## 示例

| 原始信息 | 归类 | 说明 |
|---|---|---|
| 支持手机号登录 | 业务需求 | 能力（有歧义，见确定性判断） |
| 验证码 6 位、5 分钟有效、错 3 次锁定 | 业务需求 | 规则 |
| 未注册手机号引导去注册 | 业务需求 | 逻辑分支 |
| 登录按钮是蓝色的、"好看""高级感" | 非需求（视觉） | 剥离 |
| 点击有 loading 动画、"丝滑""流畅" | 非需求（交互） | 剥离 |
| 用 Redis 存验证码、"用 XX 框架" | 非需求（技术） | 剥离（除非是业务约束，如"必须兼容老系统"） |
| 网络超时要能重试、提示"网络异常" | 业务需求 | "能不能重试"是逻辑；文案怎么排版才是设计 |
