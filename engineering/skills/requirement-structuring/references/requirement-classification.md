# 需求判定：什么算业务需求

本文件定义"哪些信息属于业务需求层、应被本 skill 处理"，哪些应被剥离。

## 核心判据

一条信息属于**业务需求**，当且仅当它在定义"做什么能力 / 遵循什么业务规则 / 有哪些逻辑分支"。

它**不**属于业务需求，当它在描述"长什么样 / 怎么交互呈现 / 怎么技术实现"。

用一句话自检：**这条信息，是在说"系统该具备什么能力或遵守什么规则"，还是在说"它该怎么呈现或怎么造出来"？** 前者是需求，后者不是。

## 分类示例

| 原始信息 | 归类 | 说明 |
|---|---|---|
| 支持手机号登录 | 业务需求 | 定义了一种能力（但有歧义，见确定性判断） |
| 验证码6位、5分钟有效、错3次锁定 | 业务需求 | 业务规则 |
| 未注册手机号引导去注册 | 业务需求 | 逻辑分支 |
| 登录按钮是蓝色的 | 非需求（视觉） | 剥离 |
| 点击有 loading 动画 | 非需求（交互） | 剥离 |
| 用 Redis 存验证码 | 非需求（技术） | 剥离，留给开发 |
| 网络超时要能重试、提示"网络异常" | 业务需求 | "能不能重试"是业务逻辑；"提示具体文案怎么排版"才是设计 |

## 交叉地带的处理

业务需求和设计之间有交叉。原则：**抓业务实质，留呈现给设计。**

例：「未注册手机号自动跳转注册页」
- 业务实质：是否存在"未注册→跳转注册"这个逻辑分支？规则是什么？→ 归需求层
- 呈现方式：跳转用什么过渡、注册页长什么样 → 留给设计，剥离

守住一条线：**你管"有没有这个逻辑分支、规则是什么"，不管"它怎么呈现给用户"。**

## 容易误判的词

以下词常意味着信息其实是"非需求"，剥离时留意：
- "好看""美观""现代感""高级" → 视觉，剥离
- "丝滑""流畅""有反馈" → 交互，剥离
- "用XX技术/框架/库" → 技术，剥离（除非这是真实的业务约束，如"必须兼容某老系统"）
