# 确定性判断：确定 / 待澄清 / 可默认

## 核心判据

标准是"信息足够让下游正确执行"，不是"信息完整"。

一条需求算**确定的**，当且仅当：任何一个合格的执行者（AI 或工程师）拿到它，做出来的结果都会落在同一个可接受的范围内——**不存在两种合理的理解会导致两种不同的实现**。

反过来，一条需求**不确定**，当它存在"合理的歧义"——两个都不傻的执行者，可能各自做出不同的东西，且双方都觉得自己对。

## 可操作的检查点

对每条需求依次过这几问：

1. **有没有歧义？** 一句话能不能有两种合理解读？
   - "支持手机号登录"→ 只能手机号 vs 新增一种？两解 → 不确定
   - "在现有邮箱登录基础上新增手机号+验证码登录"→ 歧义消失 → 趋向确定

2. **关键的输入、输出、规则明确了吗？**
   - 输入是什么、输出是什么、中间业务规则是什么。三样明确，主干就确定。
   - "验证码6位、5分钟有效、错3次锁定"→ 规则明确 → 确定
   - "做个验证码功能"→ 规则全空 → 不确定

3. **执行者需不需要替别人拍板才能往下做？**
   - 如果做到某步必须停下来替产品经理做一个本该他做的决定 → 那个点不确定。
   - 确定的标志：执行者不需要替别人拍板，只需照做。

## 三分流：确定 / 待澄清 / 可默认

```
这条需求，执行者能不能不靠猜就照做？
├─ 能 → 【确定的需求】
└─ 不能（有合理歧义）→ 看歧义性质：
    ├─ 业务逻辑歧义（功能、规则、分支、参数）→ 【待澄清】必须问
    └─ 文案 / 呈现歧义（措辞、排序、展示形式）→ 【可默认】按合理默认做、标注假设、不阻塞
```

灰色地带（如错误提示同时承载业务反馈 + 措辞）：业务部分问，措辞部分默认。拿不准归待澄清。

## 暗示歧义的信号

- 模糊动词："支持""优化""完善""改进""类似……一样" → 大概率有歧义
- 缺主语/缺宾语：没说清"对谁""改哪个"
- 量词缺失：没说"多少""多久""几次"
- 分支缺失：描述了主流程，没说"失败/异常/未命中时怎么办"
