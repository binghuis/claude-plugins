# 从 UI 草图提取业务需求

适用场景：原始需求里掺着产品顺手画的 UI（草图、示意图、"大概长这样"），尤其在没有专职设计参与的项目里。

## 核心原则

产品画的 UI，**当作"意图的参考"，不是"必须照着像素级实现的规格"**。

但它里面往往隐含真正的业务需求。你的任务是**把业务需求从图里翻译出来**，而不是把草图本身当成需求去执行。

## 两个都错的极端

- ❌ **当圣旨**：像素级照着实现。产品画的多是随手的、不专业的，照做既限制呈现专业空间，产品自己也没想让你照画的来。
- ❌ **直接忽略**：因为"UI 不是需求"就丢掉。图里藏着他没用文字写的业务信息（有哪些字段、哪些操作、信息怎么组织），丢了就丢了需求。

## 正确做法：翻译

1. **提取业务需求**：从图里读出功能元素、信息结构、操作入口，写进需求层。
   - 例：登录页草图 → 提取出"需包含手机号输入、验证码输入、获取验证码、登录、跳转注册"等功能元素（确定的需求）。

2. **标注 UI 为参考非规格**：在产出文档里写明——"附：产品提供的 UI 草图，作为布局和元素的意图参考，非最终设计规格，具体呈现以设计/前端实现为准。"

3. **不对视觉细节做澄清**：草图画得歪、颜色随便、间距不对——这些不进待澄清清单。它们是视觉层的，产品也没打算精确表达。

## 最该警惕：藏在 UI 里的隐性业务逻辑

这是产品画 UI 最容易埋的雷，也是本环节最值钱的产出。要主动揪出图里"没用文字说、但暗示了业务规则"的东西：

- 草图上小字"30秒后重新获取" → 暗含：验证码有冷却时间（一条业务规则）
- 画了"记住我"勾选框 → 暗含：需要登录态持久化；但"记多久、怎么实现"是缺口 → 待澄清
- 画了某按钮但没说点击行为 → 逻辑缺口 → 待澄清

把这些隐性业务需求提取成明确的需求条目或待澄清项。

## 关于"无专职设计"项目的提醒（不在本 skill 处理范围）

若项目没有设计参与、也没有设计规范，那"视觉呈现"是个真空地带——最后通常靠一套设计规范/组件库来保证一致性。这是"设计体系"问题，不是"需求结构化"问题。心里有数即可，**不要把它揽进本 skill 的产出**。
