# 结构化需求文档 输出 Schema

## 可追溯性要求

- 每条需求、每个待澄清项、每个开发任务都要有稳定编号（如 R1、Q1、T1）
- 标注来源：原始 / AI补充 / 已澄清确认
- 标注关联与依赖：需求之间、任务与待澄清项之间的依赖要写出来

## 文档结构

```
# 需求结构化：<功能名>

## 0. 元信息
- 来源：<原始需求出处>
- 状态：草稿 / 澄清中 / 已对齐

## 1. 确定的需求
（执行者不用猜就能照做的部分，逐条列出，每条标注来源：原始/AI补充/已澄清确认）

## 2. 待澄清清单（核心产出）
仅装会改变架构 / 范围 / 方向的 PM 拍板项；参数 / 文案 / 惯例归节 3。
每条包含：
- 问题：<哪里不清楚>
- 影响：<不澄清会导致什么>
- 候选项：<若能推测，列出可能选项>
- 归属：<谁来回答：产品/设计/技术负责人>
- 状态：待回答 / 已确认 / 已忽略

## 3. 按默认执行（不阻塞）
（参数 / 文案 / 惯例等低风险默认值。列出：默认怎么做 + 假设是什么，PM 后续 review 可改）

## 4. 边界与约束
- 范围边界：做什么 / 明确不做什么
- 前提依赖
- 需遵守的业务/技术约束

## 5. 异常与边界情况
异常 / 失败 / 空状态 / 极端输入 / 超时的业务反馈，原始需求多半未写明，必须主动补齐；写"反馈什么"，不写"长什么样"。

## 6. 拆解出的开发任务
（对齐后的需求拆成下游 skill 工作流可直接接手的结构化任务单元）

## 7. UI 参考（若有）
- 附产品 UI 草图，标注：意图参考，非最终设计规格
```

## 填好的示例

```
# 需求结构化：手机号登录

## 0. 元信息
- 来源：产品口述 + 一张登录页草图
- 状态：澄清中（第1轮）

## 1. 确定的需求
- [R1][原始] 用户可用手机号 + 验证码登录
- [R2][原始] 验证码 6 位数字，5 分钟有效
- [R3][原始] 连续输错 3 次锁定账户
- [R4][AI补充] 验证码发送有 60 秒冷却（来源：草图小字"60秒后重新获取"）

## 2. 待澄清清单
- [Q1] 问题：手机号登录是替换现有登录方式，还是新增一种？
  影响：决定整体登录架构改动量
  候选项：A 替换 / B 新增（推测 B）
  归属：产品 ｜ 状态：待回答 ｜ 关联：R1、T1
- [Q2] 问题：未注册的手机号如何处理？
  影响：缺这个分支会导致未注册用户卡死
  候选项：A 拦截提示 / B 自动引导注册
  归属：产品 ｜ 状态：待回答 ｜ 关联：T1
- [Q3] 问题：账户锁定后如何解锁？
  影响：决定有无解锁通道、是否需要额外接口
  候选项：A 超时自动 / B 客服人工 / C 用户找回流程
  归属：产品 ｜ 状态：待回答 ｜ 关联：R3、T5

## 3. 按默认执行
- [D1] 验证码输入框默认只接受数字、自动聚焦下一位（标准体验）
- [D2] 登录失败提示默认"验证码错误，请重试"（措辞）
- [D3] 账户锁定时长默认 30 分钟（行业惯例）

## 4. 边界与约束
- 范围：仅做手机号+验证码登录的业务逻辑
- 不做：第三方登录、找回密码（不在本次范围）

## 5. 异常与边界情况
- [E1] 手机号格式非法 → 提交前拦截，给出业务提示
- [E2] 验证码发送失败（网关异常）→ 允许重试，提示发送失败
- [E3] 网络超时 → 允许重试

## 6. 拆解出的开发任务
- [T1] 手机号+验证码登录接口对接（依赖：Q1、Q2 未决，暂不开工）
- [T2] 验证码发送/冷却逻辑（依赖：R2、R4，可开工）
- [T3] 错误次数计数 + 锁定逻辑（依赖：R3、D3 已定，可开工）
- [T4] 各异常分支业务处理（依赖：E1–E3，可开工）
- [T5] 解锁通道（依赖：Q3 未决，暂不开工）

## 7. UI 参考
- 附产品登录页草图，作为元素和布局的意图参考，非最终设计规格，呈现以前端实现为准
```

## 注意

- 待澄清项未解决前，**依赖它的开发任务要标注依赖并暂缓开工**（如上 T1、T3），避免在未对齐时盲目开工。
- 文档是活的：澄清推进时更新各项状态，"待澄清"逐步转入"确定的需求"；已确认的随后锁定。
