在回答之前仔细思考并一步一步来；这个问题比看上去要难。

按 `references/output-schema.md` 产出结构化文档。

## 业务 / 非业务过滤

- 需求：功能 / 业务规则 / 逻辑分支；PM 明示的交互、文案同属此类
- 非需求：视觉细节；PM 未明示的交互、文案走默认
- 混合素材：业务部分入需求清单，其余各归本位

UI 图传达的是意图，不是实现细节

| 原始素材 | 归类 |
|---|---|
| 支持手机号登录 | 需求（功能） |
| 验证码 6 位、5 分钟有效、错 3 次锁定 | 需求（规则） |
| 未注册手机号引导至注册 | 需求（分支） |
| "点击登录按钮跳转到验证码输入页" | 需求（分支：交互流程） |
| UI 图小字"30 秒后重新获取" | 需求（暗示规则 → 验证码冷却） |
| UI 图的颜色、间距、对齐偏差 | 非需求（视觉细节，不入清单也不澄清） |
| 登录失败提示"账号或密码错误" | 需求（规则：失败反馈文案） |

## 三分（确定 / 待澄清 / 可默认）

只问改变架构/范围/方向的决定，参数/数值/文案/惯例由 AI 默认

- 架构 / 范围 / 方向 PM 已明示 → 【确定】
- 架构 / 范围 / 方向 PM 没拍板 → 【待澄清】必须问
- 只缺参数 / 文案 / 惯例 → 【可默认】合理默认 + 标假设

拿不准归待澄清。
