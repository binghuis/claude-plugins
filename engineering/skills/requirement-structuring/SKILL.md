# 需求结构化（Requirement Structuring）

把产品经理用自然语言写的、形式各异的原始需求，转成结构化、可被下游开发工作流执行的文档。核心不是补全需求，而是对齐前的分流 + 循环到收敛：固化确定的需求、把代价高的歧义与缺口暴露出来交人拍板、对代价低的不确定项按合理默认执行并标注假设、人回答后增量重扫直到不再产生重大歧义。产出是承上启下的工作文档：上承原始需求，下接开发工作流。

## 适用边界

本 skill 只处理业务需求层：功能能力、业务规则、逻辑分支、边界条件。

不处理：视觉设计（颜色、布局美观）、交互呈现细节（动画、loading 样式）、技术实现方案（用什么组件/存储/接口）。这些分别属于设计、设计规范、开发决策。

原始需求里常掺着这些（尤其产品顺手画的 UI 草图）。处理原则见 `references/ui-extraction.md`：UI 当"意图参考"非"实现规格"，从中提取业务需求，不对视觉细节做澄清。

## 总体流程

```
入口判断：全新需求 ？ 还是 变更（已有结构化文档的改动）？
   │
   ├─【全新需求】→ A. 单次结构化  →  B. 澄清循环到收敛  →  C. 按业务内聚拆分
   │
   └─【变更需求】→ D. 增量处理（diff + 影响范围）→ B. 澄清循环到收敛（仅对受影响部分）
```

判断需求是"全新"还是"变更"，是第一步。两条路径的区别见下文 D 节与 `references/iterative-requirements.md`。

## A. 单次结构化（全新需求的核心）

拿到原始需求后：

### A1. 判断需求类型
- 全新需求（1.0）：从零开始，参照系是空白，只需让需求自身完整自洽。
- 迭代需求：基于现有项目代码的改动，"对错"相对于现有实现定义，需参照代码库。见 `references/iterative-requirements.md`。
- 变更需求：基于已有结构化文档的改动，走 D 节增量处理。

> 当前版本以全新需求为主干。迭代/变更依赖对"现状"（代码或已有文档）的理解，先把全新需求闭环跑通。

### A2. 剥离非需求信息
通读原始需求，分三类：
1. 业务需求 → 保留，进入下一步
2. 视觉/交互/技术信息 → 剥离；若来自 UI 草图，按 `references/ui-extraction.md` 提取其中隐含的业务需求
3. 背景/动机 → 保留作上下文，不作为可执行需求

判定标准见 `references/requirement-classification.md`。

### A3. 对每条业务需求做三分判断
判断标准见 `references/certainty-criteria.md`：
- 确定的：任何合格执行者照做，结果落在同一可接受范围，不存在导致不同实现的合理歧义。
- 待澄清的：存在合理歧义 且 做错代价高 → 必须问。
- 可默认的：存在歧义但代价低、易修改 → 按合理默认执行，标注假设，不阻塞。

第一轮就尽量把所有可识别的歧义、缺口、边界一次性揪全、揪到底，不要挤牙膏。这能把后续澄清轮数压到最少。

### A4. 识别隐性需求与边界情况
主动补出原始需求最易漏的部分：异常/失败/空状态/极端输入/超时的业务反馈（是"反馈什么"非"长什么样"）；藏在 UI 草图里的隐性业务规则；逻辑分支缺口。识别后归入"待澄清"或"可默认"。

### A5. 按 schema 产出结构化文档
按 `references/output-schema.md` 产出。

## B. 澄清循环到收敛

需求结构化不是一次性产出，而是循环到收敛的过程。

```
产出待澄清清单 → 人回答 → 把答案补回文档
   → 对"补充内容 + 其牵连到的已确定项"重新扫描
   → 还有新的"待澄清（代价高）"项吗？
        ├─ 有 → 再来一轮
        └─ 没有（只剩 确定 + 可默认）→ 收敛，停止
```

### 关键认知：澄清是收敛的，不是发散的
- 一个功能里真正会导致返工的大歧义就那么几个，第一轮基本揪光。
- 后续每轮处理的是"答案引出的次级问题"，数量越来越少。
- 通常 1–3 轮收敛。轮数不预先固定，由"还有没有重大歧义"这个终止条件决定。
- 若某需求澄清五六轮还在冒新的重大问题 → 不是流程问题，是需求本身没想清楚。此时 AI 揪出"这需求没成型"本身就是有价值的信号，应打回让产品先想清楚。

### 减少轮数的两个手段
1. 第一轮一次性揪全（见 A3）。
2. 把"可默认"项直接走默认、不进澄清清单，减少需人工介入的次数。

### 活文档
待澄清项有状态：`待回答 / 已确认 / 已忽略`。随澄清推进更新，从一堆问号逐步变成可执行需求。已确认的内容随后被"锁定"，非必要不重新审视。

## C. 按业务内聚拆分

### 默认单位：一个内聚的业务功能 = 一份结构化文档
- 不要全塞一个大文档（会失控、待澄清混在一起分不清、下游不知执行哪段）。
- 不要碎成一堆（粒度太细丢上下文、看不清全貌）。

### 拆分依据：业务内聚性，不是文档长度
判据：这些需求是否围绕同一业务目标、共享同一套上下文和规则？
- 是 → 同一份文档（哪怕长）。例："手机号登录"和"验证码冷却规则"共享上下文，不拆。
- 否 → 分开（哪怕每份短）。例："登录"和"支付"是独立业务，分两份。

### 顺序：先完整结构化，再拆
不要一上来就切块——还没搞清全貌就拆会拆错（把该在一起的逻辑切散、把未对齐的歧义带进拆分）。正确顺序：
1. 先对整个需求做结构化（A + B），形成完整认识。
2. 识别其中有几个相对独立的业务单元。
3. 若含多个独立单元 → 拆成多份文档，每份一个内聚功能（必要时加一份总览说明彼此关系）。若本身内聚 → 一份，不拆。
4. 在每份文档内部，再把它拆成"开发任务"（schema 第6部分）。

### 两层"拆"别混淆
- 业务级拆分：把大需求拆成几份功能文档 → 依据业务内聚性。
- 执行级拆分：把一份功能文档拆成几个开发任务 → 依据可执行性。

### 单份会不会越来越大？
不会。单份被业务边界框住，功能对齐后基本稳定。文档数量随功能增长（每个新功能一份），但每份小而稳。用数量增长换取单份可控。文档多了的管理问题（按模块归类、建索引）远比维护一个巨型文档容易。

### 迭代场景
迭代需求的拆分要对齐现有代码模块边界，让需求文档和代码结构对应，便于执行和影响范围分析。

## D. 变更增量处理（需求文档改动后）

产品希望定稿冻结，但需求一定会变。变更时绝不全量重跑——既浪费，又会破坏已对齐的共识（已确定项可能被重新搅乱、判断前后不一致）。

### 做法：只处理"变化的部分"和"变化所影响的部分"
1. 识别 diff：新版 vs 上一版，找出新增/删除/修改了什么。
2. 只对变化部分做结构化和澄清（走 A3/A4 + B）。
3. 评估影响范围（最易漏、最重要）：一个改动很少孤立。找出它牵连到的、原本已确定的需求，把它们从"确定"重新激活为"需重新审视"。例：原定"手机号登录"，现改为"必须先邮箱验证才能绑手机号" → 冲击原已确定项，需重新审视。
4. 未受影响的已对齐内容保持锁定，不动。

### 本质：变更处理 = 迭代需求的一种
只是迭代对象从"现有代码"变成"现有需求文档"。判断影响范围、区分受影响/不受影响——与 `references/iterative-requirements.md` 同一思路，可复用，无需另造机制。

### 前提：文档可追溯、影响可定位
要支持增量，文档不能是一团浆糊。每条需求有编号，需求间的依赖/关联可见，这样改了某条能顺着关联找到受影响的条目。（这也是 schema 里"开发任务标注依赖"的另一用途：依赖关系既给开发看，也用于改动时定位影响范围。）

### 附带价值
这套机制让需求变更从"混乱的口头改动"变成"可追溯、可评估、有边界的受控变更"。变更不可避免，但可记录、可控——这本身可作为这套范式的一个卖点。

## 关键原则（贯穿全程）

- 暴露，而非补全：让 AI 做它擅长的（识别哪里模糊），不做它不擅长的（猜真实意图）。
- 按代价分配澄清精力：核心、改错代价高的（支付、权限、数据结构）门槛高；边缘、易改的走默认。否则待澄清清单长到没法用。
- 只对业务需求负责：UI 是信息来源，不是产出对象。守住"管业务规则，不管呈现方式"这条线。
- 循环到收敛，不是一次性：澄清是有限收敛的，由终止条件决定何时停。
- 增量优于重跑：变更只处理 diff + 影响范围，锁定已对齐内容。

## 输出对机器友好
产出文档要喂给下游开发工作流（其他 skill）执行，须结构化、字段清晰、可被 AI 精准解析，而非散文式文档。格式见 `references/output-schema.md`。
