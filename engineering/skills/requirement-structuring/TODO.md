## 留到后面：迭代需求（代码库参照）

**为什么现在不做**：
1. 难点不在"写规则"，在底层的"代码库理解能力"——这是 Cursor / Claude Code 等顶尖工具都还在攻坚的能力，现在凭空设计规则是空中楼阁。
2. 策略是"借力现有 AI 编程工具"，但借哪个工具的什么能力、能力边界在哪、规则怎么搭在其上——只有真正上手试过才知道，现在设计大概率是错的。
3. 顺序上，迭代是"在全新需求底座上加一层"。底座（主干）没验证稳之前搭上层，是在未夯实的地基上盖二楼。

**现在的状态**：`references/iterative-requirements.md` 已做"清醒占位"——锁定了方向（为什么难、借力别自造、要额外产出什么）。**设计不用再补，占位已够。**

**启动条件**（两个都满足再做）：
- [ ] 主干已用真实全新需求验证稳定
- [ ] 已真正上手试过迭代场景，摸清了可借力的工具能力边界

**届时要做**：
- [ ] 选定并接入现有 AI 编程工具的代码库理解能力
- [ ] 实现"与现有实现的关系"分析
- [ ] 实现"影响范围 / 改动面"分析
- [ ] 实现"基于现有实现的待澄清项"识别（需求与现状对不上的冲突点）
- [ ] 迭代需求的拆分对齐现有代码模块边界
