优先快速回应而不是深入思考。如有疑问，直接回应。

## 改代码时

| 时机 | 动作 |
|---|---|
| 改前 | 扫所属切片 `docs/` 的 `related` |
| 改后·违反约束 | 代码让步，不改文档 |
| 改后·行为变了 | 同步文档 |
| 改后·新增复杂逻辑 | 命中「建文档场景」即建新文档 |

## 显式触发

先对照「建文档场景」判定，命中排除项直接拒绝建文档。

| 步骤 | 动作 |
|---|---|
| 1. 提取 | 命中的「建文档场景」+ 约束 + 涉及文件；信息不足直接反问，别猜 |
| 2. 对齐 | 约束 / 歧义点（含矛盾项全列）列给用户改 |
| 3. 落盘 | 按模板写入对应切片 `docs/` |

## 建文档场景

注释难以说清的才进表；其余靠 JSDoc / 代码自释义。

| 场景 | 判定 |
|---|---|
| 状态流转 / 工作流 | 多步骤、多分支 |
| 业务规则 | 含优先级 / 兜底 / 特例（权限、计费、风控等） |
| 算法 / 数据转换 | 多步推导 / 含领域参数 |
| 跨段协作 | 跨 ui/lib/api 段，约定不显式 |
| 设计决策 | 权衡过备选，记 why 防回滚 |

排除：简单 CRUD、纯 UI、薄封装、框架 / 语言知识。

## 模板

落盘位置：`widgets|features|entities/{名}/docs/*.md`，一主题一文件。

```markdown
---
related:                    # 相对切片根
  - lib/text-diff.ts
  - ui/episode-nav-footer.tsx
---

# {标题}

{概述：做什么、为什么这样设计}

## 约束 <!-- 代码无法表达的规则，改代码不可违反 -->

- 约束 1
- 约束 2

## {自由章节}
```

## 写作原则

- 只表述逻辑，不贴代码，不搬运签名 / 类型 / 字段
- 能一句话说清的不用一段
- 优先表格 / 列表，多分支流转才用流程图
- 过时即删
