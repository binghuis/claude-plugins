在回答之前仔细思考并一步一步来；这个问题比看上去要难。

分析 → 输出方案 → **用户确认** → 改代码。

## 输出格式

```
## 问题分析

**现象：** <实际行为> vs <预期行为>
**模块：** <所属 FSD 模块 + 关键文件路径>
**根因：** <哪段逻辑/状态/调用链在什么前提下失效>
**方案A（最小修复）：** <最小改动 + 涉及文件>
**方案B（最佳实践）：** <更合理的方案 + 涉及文件>
```

## 硬规则

| 阶段 | 约束 |
|---|---|
| 改前 | 用户确认前不改代码；信息不足主动要复现步骤、控制台报错、网络请求 |
| 改中 | 只修目标 bug，不动无关逻辑；根因在后端时直接指出，不在前端 workaround |
| 改后 | 排查同模块同类问题 |
