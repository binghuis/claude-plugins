# 注释规则

## 需要注释

- **业务规则** — 仅靠读代码无法理解的"为什么"
- **非显而易见的意图** — 看似奇怪但有原因的写法、绕过已知问题的 workaround
- **关键决策** — 为什么选 A 不选 B，约束或权衡
- **边界条件** — 特殊值、阈值、魔法数字的含义
- **复杂算法/流程** — 多步骤逻辑的整体思路概述
- **TODO** — `// TODO: 描述`

## 不需要注释

- 名字已自解释的代码、类型字段
- 框架/语言标准用法（`useEffect`、`.map()` 等）
- 纯样式、`import`、常规 CRUD

## 风格

- 中文，简洁直白，说"为什么"不说"做了什么"
- 行内用 `//`，函数/对外接口用 `/** */`
- 不写装饰分隔线、ASCII art、无意义 header

## 触发方式

**A. 写完代码后自动评估（默认）**

刚写或改完代码时，对照上面的清单评估改动点。命中就补，没命中就跳过，不要凑数。改动已在上下文中，无需 `git diff`。

**B. 用户主动调用**

用户明确要求"补注释 / 审查注释"时：

1. `git diff HEAD` 查看本次变更
2. 逐文件按清单识别并补注释，不为覆盖率硬加
