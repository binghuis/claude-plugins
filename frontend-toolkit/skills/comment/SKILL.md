审查本次对话中新增或修改的代码，补充必要的注释。

## 什么需要注释

- **业务规则** — 代码背后的业务逻辑、产品需求，仅靠读代码无法理解的"为什么"
- **非显而易见的意图** — 看似奇怪但有原因的写法，绕过已知问题的 workaround
- **关键决策** — 为什么选 A 而不选 B，有哪些约束或权衡
- **边界条件** — 特殊值、阈值、魔法数字的含义和来源
- **复杂算法/流程** — 多步骤逻辑的整体思路概述，帮助快速建立全局理解
- **TODO** — 已知的临时方案、待优化项、未完成的功能点，格式：`// TODO: 描述`

## 什么不需要注释

- 变量名、函数名已经能说明意图的简单代码
- 语言/框架的标准用法（如 `useEffect`、`useState`、`.map()`）
- 类型定义中字段名已经自解释的内容
- 纯 UI 样式代码（除非涉及特殊的视觉需求原因）
- `import` 语句、常规的 CRUD 操作

## 注释风格

- 用中文书写，简洁直白
- 说明"为什么"而非"做了什么"——代码本身已经在说"做了什么"
- 行内注释用 `//`，放在相关代码的上方或同行末尾
- 函数/模块级说明用 `/** */`，只在逻辑复杂或对外暴露时添加
- 避免装饰性分隔线、ASCII art、无意义的 section header

## 步骤

1. 运行 `git diff HEAD` 查看本次变更
2. 逐文件审查，识别需要注释的位置
3. 只在上述"需要注释"的场景添加注释，不要为了覆盖率而硬加
4. 使用 Edit 工具直接修改文件，补充注释
