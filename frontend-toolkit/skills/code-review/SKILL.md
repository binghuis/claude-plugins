在回答之前仔细思考并一步一步来；这个问题比看上去要难。

只纠错，不提升。

| 命中（报） | 越界（不报） |
|---|---|
| 线上会出错、数据损坏、明显性能问题 | 风格、命名、格式、注释多寡、import 顺序 |
| 特定场景下逻辑走错 | 是否拆函数、抽象是否优雅、可读性 |
| 直接引发错误的设计缺陷 | 单纯"不够优雅"、未来可维护性顾虑 |

## 变更来源

| 情况 | 命令 |
|---|---|
| 工作区或暂存区有未提交改动 | `git diff HEAD` |
| 否则 | `git show HEAD` |

仅看 diff 不够时，主动读调用方、被调用方、共享状态、相关类型。

## 维度

| 维度 | 典型隐患 |
|---|---|
| 数据流 | 竞态条件、缓存与实际数据不一致、异步顺序依赖、状态更新丢失 |
| 逻辑 | 条件分支遗漏、同一概念多处判断标准不同、默认值掩盖空数据 |
| 权限 | 权限判断遗漏或不一致、角色/状态组合未覆盖、不同入口走了不同的权限逻辑 |
| 设计 | 状态放置错位导致数据源不一致、职责错配导致行为与预期不符（仅限直接引发错误的部分） |

## 输出

用通俗口语；非用术语不可时紧跟一句人话解释。

按严重程度倒序列出：🔴 必修（线上会出错或数据损坏）/ 🟡 应修（特定场景下出错）/ 🟢 留意（边角场景）。

每条隐患用如下模板：

```
#### [标题] — [数据流|逻辑|权限|设计] 🔴
**问题**：用户或业务层面看到的异常结果，一句话（如"切换 tab 后列表偶尔显示上一个 tab 的数据"）；不写设计、不贴代码、不讲实现
**根因**：用业务语言说清"该发生什么、实际发生了什么、为什么会错"，不写技术细节（如"用户切到新 tab 时，旧 tab 的数据请求还没回来，回来后又把界面覆盖了"，而不是"useEffect 的 race condition"）
**模块**：所属 FSD 模块 + 关键文件路径
**复现场景**：用户具体怎么操作能稳定看到这个问题（如"在订单页快速点击两次不同的 tab"）
**解决方案**：业务上要怎么处理才对，一两句话讲清思路（如"切 tab 时丢弃上一次未完成的请求结果"），不贴代码
```

末尾：总结（🔴×a 🟡×b 🟢×c）+ Top 3 必修。无问题直接说"无问题"。

## 硬规则

- 说不清"什么场景会出事"就不报
