# Code Review

只找核心隐患 — 会导致业务异常或严重性能问题的代码。不报风格、命名、格式等边角问题。

## 变更来源

- 工作区或暂存区有未提交改动：`git diff HEAD`
- 否则：最近一次 commit `git show HEAD`

仅看 diff 不够时，主动读调用方、被调用方、共享状态、相关类型——隐患常藏在 diff 之外的耦合里，不要凭 diff 猜测。

## 维度

1. **数据流** — 竞态条件、缓存与实际数据不一致、异步顺序依赖、状态更新丢失
2. **逻辑隐患** — 条件分支遗漏、逻辑不一致（同一概念多处判断标准不同）、默认值掩盖的空数据问题
3. **权限与状态** — 权限判断遗漏或不一致、角色/状态组合未覆盖、不同入口走了不同的权限逻辑
4. **设计问题** — 职责不清导致修改连锁反应、状态放置不当导致数据源不一致、组件耦合导致不可预测行为

## 输出

按严重程度倒序列出。严重程度：🔴 必修（线上会出错或数据损坏）/ 🟡 应修（特定场景下出错）/ 🟢 留意（边角场景）。

```
#### [标题] — [数据流|逻辑|权限|设计] 🔴
**问题**：会引发什么后果（不是描述"做了什么"）
**位置**：`file:line`
**触发条件**：什么场景会真实暴露（如"用户快速切换 tab 时"）
```

末尾：总结（🔴×a 🟡×b 🟢×c）+ Top 3 必修。无问题直接说"无问题"，不写"建议关注"类废话。

## 规则

- 说不清"什么场景会出事"的不报
- 只指问题，不写修复代码（让用户决定怎么改）
- 跨文件影响必须读相关文件确认，不靠猜
- 不报：风格、命名、格式、注释多寡、import 顺序、是否拆函数等主观偏好
