在回答之前仔细思考并一步一步来；这个问题比看上去要难。

只纠错，不提升。命中即报，越界不报。

| 命中（报） | 越界（不报） |
|---|---|
| 线上会出错、数据损坏、明显性能问题 | 风格、命名、格式、注释多寡、import 顺序 |
| 特定场景下逻辑走错 | 是否拆函数、抽象是否优雅、可读性 |
| 直接引发错误的设计缺陷 | 单纯"不够优雅"、未来可维护性顾虑 |

## 变更来源

| 情况 | 命令 |
|---|---|
| 工作区或暂存区有未提交改动 | `git diff HEAD` |
| 否则 | `git show HEAD` |

隐患常藏在 diff 之外的耦合里。仅看 diff 不够时，主动读调用方、被调用方、共享状态、相关类型。

## 维度

| 维度 | 典型隐患 |
|---|---|
| 数据流 | 竞态条件、缓存与实际数据不一致、异步顺序依赖、状态更新丢失 |
| 逻辑 | 条件分支遗漏、同一概念多处判断标准不同、默认值掩盖空数据 |
| 权限与状态 | 权限判断遗漏或不一致、角色/状态组合未覆盖、不同入口走了不同的权限逻辑 |
| 设计 | 状态放置错位导致数据源不一致、职责错配导致行为与预期不符（仅限直接引发错误的部分） |

## 输出

按严重程度倒序列出：🔴 必修（线上会出错或数据损坏）/ 🟡 应修（特定场景下出错）/ 🟢 留意（边角场景）。

每条隐患用如下模板：

```
#### [标题] — [数据流|逻辑|权限|设计] 🔴
**问题**：用户或业务层面看到的异常结果，一句话（如"切换 tab 后列表偶尔显示上一个 tab 的数据"）；不写设计、不贴代码、不讲实现
**根因**：哪段逻辑/状态/调用链在什么前提下失效
**模块**：所属 FSD 模块 + 关键文件路径
**触发条件**：什么场景会真实暴露（如"用户快速切换 tab 时"）
```

末尾：总结（🔴×a 🟡×b 🟢×c）+ Top 3 必修。无问题直接说"无问题"。

## 硬规则

- 说不清"什么场景会出事"就不报
- 只指问题，不写修复代码
- "问题"写后果，技术细节归"根因"
