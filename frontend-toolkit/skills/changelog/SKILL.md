优先快速回应而不是深入思考。如有疑问，直接回应。

基于本次对话的代码变更，生成 changelog 条目并追加到 `CHANGELOG.md`。

## 步骤

1. 取 diff（工作区未暂存的一律忽略）：

    | 情况 | 命令 |
    |---|---|
    | 暂存区有改动 | `git diff --cached` |
    | 暂存区为空 | `git show HEAD` |

2. 过滤：只保留用户可感知的变化，跳过纯工程改动（重构、依赖升级、CI、工具链）以及 `CHANGELOG.md`、`node_modules`、`dist`、`.env` 等非源码文件。

3. 归类：

    | 类型 | 含义 |
    |---|---|
    | 新增 | 新功能、新能力 |
    | 优化 | 体验改进、交互优化、视觉升级 |
    | 修复 | 问题修复 |
    | 移除 | 下线功能 |

4. 每条写一句简洁中文（≤ 80 字），用户/产品视角，口吻为产品发布公告，说明做了什么、带来什么价值：

    | ✓ 用户语 | ✗ 技术语 |
    |---|---|
    | 工作流进度条增加彩色连接线，完成状态一目了然 | 重构 WorkflowProgressBar 改用 flex 布局 |
    | 表单提交失败时给出清晰错误提示 | onSubmit catch 块补抛错误 |

    用业务名词描述结果，跳过文件名、组件名、变量名、CSS 类名以及"重构/布局/样式/组件"等开发术语。

5. 读取 `CHANGELOG.md`，**完整读完当天日期块**——从 `## YYYY-MM-DD` 一直读到下一个 `## ` 或文件结尾，确保对结构的判断基于完整内容。

6. 模板（**仅用于新建日期块**；已有日期块按第 7 步合并）：

    ```markdown
    ## YYYY-MM-DD

    ### 新增
    - 描述

    ### 优化
    - 描述

    ### 修复
    - 描述

    ### 移除
    - 描述
    ```

    空分类不写。日期取本地当天。

7. **逐层合并**，同一日期块内保持 `###` 分类唯一：

    | 层级 | 规则 |
    |---|---|
    | 日期层 | 当天已有 `## YYYY-MM-DD` 则沿用；否则在 `# Changelog` 后插入新日期块 |
    | 分类层 | 同名 `###` 存在 → 追加到该分组**最后一个条目之后**；不存在 → 新建，按 `新增 → 优化 → 修复 → 移除` 顺序插入 |

    写入前自检：扫当天日期块所有 `### ` 标题，发现重名（历史遗留）一并合并。

8. 写入 `CHANGELOG.md` 并 `git add CHANGELOG.md`。
