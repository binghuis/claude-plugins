基于本次对话的代码变更，生成 changelog 条目并追加到 `CHANGELOG.md`。

## 步骤

1. 运行 `git diff --cached` 查看**已暂存**的代码变更（工作区未暂存的改动一律忽略）。
   - 若暂存区为空，运行 `git show HEAD`，基于最新一次提交生成条目。
2. 将变更归类到以下一个或多个类型：
   - **新增** — 新功能、新能力
   - **优化** — 对已有功能的体验改进、交互优化、视觉升级
   - **修复** — 问题修复
   - **移除** — 下线的功能
3. 为每条变更写一句简洁中文描述（≤ 80 字）：
   - 采用**用户/产品**视角：说明做了什么、带来什么价值。
   - **不要**出现文件名、组件名、变量名、CSS 类名等技术细节。
   - **不要**使用"重构""布局""样式""组件"等开发术语，换用用户能理解的说法。
   - 示例：✅ "工作流进度条增加彩色连接线，完成状态一目了然" ❌ "重构 WorkflowProgressBar 改用 flex 布局"
4. 读取 `CHANGELOG.md`，**必须完整读完当天日期块**——从 `## YYYY-MM-DD` 读到下一个 `## ` 或文件结尾，禁止只看开头几行就假定结构完整。
5. 模板格式（**仅用于新建日期块**；已有日期块时按第 6 步合并，不要整体套用模板）：

    ```markdown
    ## YYYY-MM-DD

    ### 新增
    - 描述

    ### 优化
    - 描述

    ### 修复
    - 描述

    ### 移除
    - 描述
    ```

    空分类不写。
6. **逐层合并**，严禁同一日期块内出现重复的 `###` 分类：
    - **日期层**：当天已有 `## YYYY-MM-DD` 则沿用；否则在 `# Changelog` 后插入新日期块。
    - **分类层**：对每个新条目，先在当天日期块内查找同名 `### 新增/优化/修复/移除`：
        - 已存在 → 追加到该分组**最后一个条目之后**（不要插在分组开头，避免视觉错位）。
        - 不存在 → 新建分类，按 `新增 → 优化 → 修复 → 移除` 顺序插入日期块。
    - 写入前自检：扫一遍当天日期块内的 `### ` 标题，若发现重名（历史遗留），本次一并合并修正。
7. 写入 `CHANGELOG.md`。
8. 运行 `git add CHANGELOG.md` 将更新加入暂存区。

## 规则

- 日期：使用本地当天
- 语言：中文
- 口吻：产品发布公告，简洁、用户友好
- 对用户无感的纯工程改动（代码重构、依赖升级、CI 配置、开发工具）不记录
- 不记录对 `CHANGELOG.md` 本身的修改
- 不记录 `node_modules`、`dist`、`.env` 等非源码变更
