优先快速回应而不是深入思考。如有疑问，直接回应。

1. 跑 `git diff --cached`；输出为空直接退出。

2. 过滤：保留产品/测试能从界面验证的变化；跳过：

    | 类别 | 例 |
    |---|---|
    | 纯工程改动 | 重构、依赖升级、CI、工具链 |
    | 非源码文件 | `CHANGELOG.md`、`node_modules`、`dist`、`.env` 等 |

3. 归类：

    | 类型 | 含义 |
    |---|---|
    | 新增 | 新功能、新能力 |
    | 优化 | 体验改进、交互优化、视觉升级 |
    | 修复 | 问题修复 |
    | 移除 | 下线功能 |

4. 每条按"在哪里做什么 → 会看到什么"写：

    | ✓ 可验证 | ✗ 不可验证 |
    |---|---|
    | 工作流进度条在已完成步骤之间显示绿色连接线 | 重构 WorkflowProgressBar 改用 flex 布局 |
    | 表单提交失败时在输入框下方以红字提示失败原因 | onSubmit catch 块补抛错误 |
    | 批量删除若包含无权限条目，自动跳过这些条目，其余正常删除（此前整批失败） | 批量删除遇到无权限条目时自动 skip |

    | 规则 | 说明 |
    |---|---|
    | 一动作一条 | 拖动、删除、保存各占一条 |
    | 标注行为变更 | 用"（此前 XX）"注明改动前的表现，让测试看出回归边界 |
    | 写场景不写规则 | 描述用户操作和肉眼可见的结果，不写"校验/判定/逻辑/匹配规则" |
    | 用界面/动作词 | 不写组件名、变量名、CSS 类、"重构/布局/样式" |

5. 读 `CHANGELOG.md`：完整读完当天日期块（从 `## YYYY-MM-DD` 到下一个 `## ` 或文件尾）。第 7 步做分类合并，需全量上下文。

6. 模板（仅用于新建日期块；已有日期块按第 7 步合并）：

    ```markdown
    ## YYYY-MM-DD

    ### 新增
    - 描述

    ### 优化
    - 描述

    ### 修复
    - 描述

    ### 移除
    - 描述
    ```

    空分类省略；日期取本地当天。

7. 逐层合并，同一日期块内保持 `###` 分类唯一：

    | 层级 | 规则 |
    |---|---|
    | 日期层 | 当天已有 `## YYYY-MM-DD` 则沿用；否则在 `# Changelog` 后插入新日期块 |
    | 分类层 | 同名 `###` 存在 → 追加到该分组最后一个条目之后；不存在 → 新建，按 `新增 → 优化 → 修复 → 移除` 顺序插入 |

    写入前扫当天块所有 `### ` 标题，重名一并合并（防历史遗留分裂）。

8. 写入 `CHANGELOG.md` 并 `git add CHANGELOG.md`。
