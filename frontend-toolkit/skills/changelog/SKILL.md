根据本次对话中的代码变更，生成 changelog 条目并追加到 `CHANGELOG.md`。

## 步骤

1. 运行 `git diff HEAD` 和 `git status` 查看所有变更（包括暂存和未暂存）。
   - 如果没有变更，运行 `git log --oneline -10` 查看最近提交，基于最新一次提交生成。
2. 分析变更内容，归类为以下类型之一或多个：
   - **新增** — 新功能、新能力
   - **优化** — 对已有功能的体验改进、交互优化、视觉升级
   - **修复** — 问题修复
   - **移除** — 下线的功能
3. 为每个变更写一条简洁的中文描述（一句话，不超过 80 字）。
   - 站在**用户/产品**视角描述：说明功能做了什么、对用户有什么价值。
   - **不要**出现文件名、组件名、变量名、CSS 类名等技术细节。
   - **不要**使用"重构""布局""样式""组件"等开发术语，改用用户能理解的表达。
   - 示例：✅ "工作流进度条增加彩色连接线，完成状态一目了然" ❌ "重构 WorkflowProgressBar 改用 flex 布局"
4. 读取 `CHANGELOG.md`，在最顶部（`# Changelog` 标题之后）插入新条目，格式如下：

```markdown
## YYYY-MM-DD

### 新增
- 描述

### 优化
- 描述

### 修复
- 描述

### 移除
- 描述
```

5. 只保留有内容的分类，空分类不写。
6. 如果今天已有 `YYYY-MM-DD` 条目，合并进去而非新建。
7. 写入 `CHANGELOG.md`。
8. 运行 `git add CHANGELOG.md` 把更新加入暂存区。

## 规则

- 日期使用当天日期
- 语言：中文
- 面向产品发布公告的口吻，简洁、用户友好
- 纯工程改动（代码重构、依赖升级、CI 配置、开发工具）如果对用户无感知则不记录
- 不要记录对 CHANGELOG.md 本身的修改
- 不要记录 node_modules、dist、.env 等非源码变更
