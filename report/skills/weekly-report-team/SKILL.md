## 1. 数据盘点 + 聚成交付物

自动拉不到的（对照上周、定性问题/风险）问用户或确认，别编。

**git · 代码产出**
```text
git log --no-merges --since="<周一> 00:00" --until="<周日> 23:59:59" --format="%an | %s"
```

**飞书数据** — 拉数前用 Skill 调 `feishu:lark-io`（文档 / 日程）、`feishu:feishu-project`（需求 / 排期）取调用链；按源填团队参数：

- **日程 / 面试**（列日程）— 时间窗＝本周起止
- **需求**（查需求）— project_key=ccu47f；MQL：
  ```sql
  -- 在做
  SELECT `name`, `work_item_status` FROM `ccu47f`.`需求`
  WHERE `业务线` = '翻译中台'
    AND `work_item_status` != '已上线' AND `work_item_status` != '已结束' AND `work_item_status` != '已终止'
  -- 本周上线
  SELECT `name` FROM `ccu47f`.`需求`
  WHERE `业务线` = '翻译中台' AND status_time('已上线') between '<周一>' and '<下周一>'
  ```
- **排期 / 工时**（查排期）— project_key=ccu47f；user_keys=宋秉徽、赵强、陈文帆；本周起止（YYYY-MM-DD）
- **版本测试报告**（读文档正文）— 标题＝"翻译中台 V<版本> 测试报告"；选取规则＝标题含目标版本号的「系统测试报告」那条

**上周计划** — 读上一份周报计划段对完成情况

聚成交付物：碎任务 → 模块（45 提交 → 5 模块），不堆流水账；过价值测试"那又怎样、谁在乎"，答不上就删。

## 2. 写周报

按价值定详略与开头放什么：问题 / 需上级拍板 → 开头单列，交付（对照上周计划）→ 正文主体，团队管理 + 下周计划 → 次之，活动量（提交 / 工时 / 会议）→ 附录当证据。

```markdown
# 周报 · <团队> · <起>–<止>

<结论行：团队主线 + 对照上周计划（要做 N→做完 M、延期 K 因 X）+（如有）需上级拍板 / 系统性风险；直接写，不加「一句话结论」小标题>

## 1、业务线版本迭代

### 1.1 <项目> · 版本 x.x.x：开发中 / 已评审 / 已提测 / 已上线
N 个需求，M 人，排期 起–止
- **<负责人>**：<交付物 + 结果> — <进度 / 状态>
- **<负责人>**：…

（多项目：每项目一个 ### 子标题，格式同上，版本线独立）

## 2、团队管理事项
工作负荷（不写"满负荷"，写几人扛几条并行线 + 产出，让读者自得出"该补人"）/ 面试（场次 + 候选人 + 轮次）/ 技术架构决策 / 交接 / 招聘。

## 3、问题 / 风险 / 需要的支持
- **<问题 / 需拍板决策>**：影响 <…> → 已试 <…> → 需要 <谁> 在 <何时> 给 <什么>；系统性风险（单点依赖 / 产能透支 / 交接缺位）带 2–3 个拍板选项（补人 / 调优先级 / 定归属）
- 无（如确无）

## 4、本周计划 / 下周计划
1. **<驱动工作>** — 完成标准：<可验证的完成定义>

## 附：数据 / 协作
- 提交 N（feat/fix/refactor）｜评审 / 分享 / 面试 …
```

**写时约束：**

- 一件事说一次 — 每条事实/风险只写在最该出现处（别按人讲完再按主题重述、别重列上周存量、在解风险不复读、只在有新变化时提）；短靠去重、不压平结构，别把人名塞括号。
- 无市场腔 — 数字带基线（"800→560ms"不是"快了"、进度写几成不写"已基本完成"）；删形容词（高质量/顺利/完美）；禁 emoji；禁套话（继续推进/护航/观察/跟进）；没数字写定性别编。
- 决策显性 — 需上级拍的进开头/问题段，自己拍的关键取舍也写。
- 不留脚手架 — 缺的交付前问用户，git 没中文名留原值别臆造。
- 飞书排版 — 别用宽 md 表格塞长文本（飞书列宽挤成竖排）；多人/多项目用「人名小标题+列表」。
