# 团队周报

## 1. 拉数据

- 代码产出 — git 拉全员按 author 分组，按模块聚成交付物、提炼到价值：
  ```
  git log --since="<周一> 00:00" --until="<周日> 23:59:59" --pretty=format:"%an | %s" | grep -v "| Merge"
  ```
- 日程 / 面试 — lark-mcp 用户身份：`calendar_v4_calendar_primary {}` `useUAT:true` 取 `calendar_id` → `calendar_v4_calendarEvent_list` 传 `path.calendar_id` + `params.{start_time,end_time,page_size:100}`（面试 `description` 含候选人 / 职位 / 轮次）。时间窗 `date -j -f '%Y-%m-%d %H:%M:%S' '<日> 00:00:00' +%s`。token 过期（报 `user_access_token invalid`）就重调一次该工具拿 `localhost:3000/authorize` 链接给用户点开授权（本地回调服务只活 60 秒、超时再调一次重拉，可无限重试）；`feishu-project` 是另一套独立授权。
- 需求 / 版本 / 排期 — `feishu-project` MCP（缺陷 / 需求 / 用例）；评审文档 `docx_v1_document_rawContent` / 搜 `docx_builtin_search`（中大型需求贴链接）。
- 上周计划 — 读上一份周报计划段对完成情况。

拉不到的（对照上周、问题、版本号 / 需求量 / 排期 / 链接、面试数据）问用户或确认，别编。

## 2. 聚成交付物 + 对照上周

原始数据加工成交付物，不堆流水账：

- 拔到交付物 + 价值 — 碎任务聚成交付物（45 提交 → 5 模块），写到"带来什么结果"那层；每条自问"那又怎样、谁在乎"，答不上就删。
- 对照上周 — 开头跟上周计划对：说要做 N → 做完 M、延期 K（因 X），第一次没上周计划就下周起生效；正文写新进展，别把存量重列一遍。
- 单轴叙述，一件事说一次 — 业务线按「项目 → 人名小标题 → 列表」；别两套索引（按人讲完别再按主题重述同一批事），一条事实 / 风险只在最该出现处写一次。短靠去重不靠压平结构，别把人名挤进括号；目标 30 秒读完开头、2 分钟读完全文。
- 负载与风险用事实 — 不写"满负荷 / 忙不过来"，用"几人扛几条并行线 + 产出"让读者自得出"该补人"。系统性风险（单点依赖 / 产能透支 / 交接缺位）放开头 + 2–3 个拍板选项（补人 / 调优先级 / 定归属与时间窗）；在解的不复读，只在有新变化（到岗 / 归属定 / 卡住排期）时提，常态把招聘写进管理段。

## 3. 套模板成稿

按价值定详略与开头放什么：问题 / 需上级拍板 → 开头单列，交付（对照上周计划）→ 正文主体，团队管理 + 下周计划 → 次之，活动量（提交 / 工时 / 会议）→ 附录当证据。

```markdown
# 周报 · <团队> · <起>–<止>

<开头结论行：本周团队主线 +（如有）需上级拍板决策；直接写，不加「一句话结论」小标题>

## 1、业务线版本迭代

### 1.1 <项目> · 版本 x.x.x：开发中 / 已评审 / 已提测 / 已上线
总共 N 个需求，投入 M 人，排期 起 – 止，中大型需求：<名称>（贴链接）
- **<负责人>**：<关键模块 / 功能> — <进度 / 状态>
- **<负责人>**：…

（多项目：每个项目一个 ### 子标题，按上格式追加，各项目版本线独立）

## 2、团队管理事项
工作负荷（几人扛几条并行线 + 产出）/ 面试（场次 + 候选人 + 轮次）/ 技术架构决策 / 交接。初期占比小也从有写起，逐步变大。

## 3、问题 / 风险 / 需要的支持
- **<问题 / 需拍板决策>**：影响 <…> → 已试 <…> → 需要 <谁> 在 <何时> 给 <什么>
- 无（如确无）

## 4、本周计划 / 下周计划
1. **<驱动工作>** — 完成标准：<可验证的完成定义>

## 附：数据 / 协作
- 提交 N（feat/fix/refactor）｜评审 / 分享 / 面试 …
```

## 4. 成稿自检

- 结论先行 + 决策显性 — 开头一句答"团队这周顺不顺 + 有没有要上级拍板的"；团队自己拍的关键取舍写出来，需上级拍的提到开头 / 问题段。
- 数字带基线和变化 — "p95 800→560ms"不是"快了"；进度写几成 / 剩什么，不写"已基本完成"；删形容词（高质量 / 顺利 / 完美）；没真实数字就写诚实定性，别编。
- 计划段只列驱动工作 + 完成标准 — 每条带可验证的完成定义；禁"继续推进 / 护航 / 观察 / 跟进"等被动待命，真没活只写真有的。周一复盘上周写「本周计划」，仅周五收尾本周才叫「下周计划」。
- 成稿格式 — 段标题按模板编号（1 / 2 / 3 / 4，业务线下 1.1 / 1.2），不写「一、二、三」；不留脚手架（缺的交付前问用户填，git 查不到中文名保留原值别臆造）；版本状态用文字（开发中 / 已评审 / 已提测 / 已上线）不用 emoji（🟢🟡🔴）；正文一律不用 emoji（含 ⚠️、✅）；格式周周一致（连续 3 周"测试中" = 红灯）。
- 飞书排版 — 别用宽 markdown 表格塞长文本（飞书列宽会挤成竖排）；多人 / 多项目用「人名小标题 + 要点列表」，非要表格就在飞书插原生表格手填短字段，别从 markdown 粘。
